# Bugfix Requirements Document

## Introduction

The labelize ZPL renderer produces text output that differs from expected testdata images (generated by Labelary) in two areas: text positioning (coordinates) and text orientation (rotation handling). The `get_text_top_left_pos()` function in `renderer.rs` applies incorrect vertical/horizontal offsets when computing the top-left drawing position for text fields, and the `DrawerState::update_automatic_text_position()` function in `drawer_state.rs` advances the automatic position cursor incorrectly for rotated text. Additionally, text glyphs are never actually rotated — only their positions are adjusted — causing orientation-dependent rendering to diverge from the reference output.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN text is rendered with `^FO` (field origin, `calculate_from_bottom=false`) and orientation is Normal THEN the system adds an incorrect `3h/4` vertical offset to the y-coordinate, shifting text downward from the expected position

1.2 WHEN text is rendered with `^FO` and orientation is Rotated90 THEN the system adds an incorrect `h/4` horizontal offset to the x-coordinate, shifting text rightward from the expected position

1.3 WHEN text is rendered with `^FO` and orientation is Rotated180 THEN the system adds an incorrect `h/4` vertical offset to the y-coordinate and shifts x by the full text width, misplacing text from the expected position

1.4 WHEN text is rendered with `^FO` and orientation is Rotated270 THEN the system adds an incorrect `3h/4` horizontal offset to the x-coordinate and shifts y by the full text width, misplacing text from the expected position

1.5 WHEN text is rendered with `^FT` (field typeset, `calculate_from_bottom=true`) and any orientation THEN the system computes the top-left position using multiline offsets but does not correctly transform the bottom-referenced y-coordinate to a top-left coordinate, resulting in vertical misplacement

1.6 WHEN text is rendered with any non-Normal orientation (Rotated90, Rotated180, Rotated270) THEN the system only adjusts the drawing position but does not rotate the actual glyph rendering, causing text to appear in the wrong direction compared to the expected output

1.7 WHEN multiple text fields use automatic positioning via `^FT` without explicit coordinates THEN the system advances the auto-position cursor by the measured text width, but the width measurement and direction of advancement per orientation may be incorrect, causing subsequent fields to be misplaced

### Expected Behavior (Correct)

2.1 WHEN text is rendered with `^FO` and orientation is Normal THEN the system SHALL place text at the field origin coordinates without adding fractional font-height offsets, matching the Labelary reference position

2.2 WHEN text is rendered with `^FO` and orientation is Rotated90 THEN the system SHALL compute the top-left drawing position so that the rotated text baseline aligns with the Labelary reference position

2.3 WHEN text is rendered with `^FO` and orientation is Rotated180 THEN the system SHALL compute the top-left drawing position so that the inverted text aligns with the Labelary reference position

2.4 WHEN text is rendered with `^FO` and orientation is Rotated270 THEN the system SHALL compute the top-left drawing position so that the rotated text baseline aligns with the Labelary reference position

2.5 WHEN text is rendered with `^FT` and any orientation THEN the system SHALL correctly transform the bottom-referenced coordinate to a top-left drawing coordinate, accounting for font height and multiline offsets, matching the Labelary reference position

2.6 WHEN text is rendered with any non-Normal orientation THEN the system SHALL rotate the actual glyph output (or render onto a buffer and rotate the buffer) so that text appears in the correct direction matching the Labelary reference output

2.7 WHEN multiple text fields use automatic positioning via `^FT` THEN the system SHALL advance the auto-position cursor correctly per orientation, so that subsequent fields are placed at the expected positions matching the Labelary reference

### Unchanged Behavior (Regression Prevention)

3.1 WHEN text is rendered with `^FO` and Normal orientation using simple single-line content THEN the system SHALL CONTINUE TO render readable text at approximately the specified coordinates (within the current tolerance)

3.2 WHEN barcodes are rendered with any orientation THEN the system SHALL CONTINUE TO rotate barcode images correctly using the existing `rotate_90`/`rotate_180`/`rotate_270` functions

3.3 WHEN graphic boxes, circles, and diagonal lines are rendered THEN the system SHALL CONTINUE TO render them at their specified positions without any change

3.4 WHEN text is rendered with `^FB` (field block) word wrapping and alignment THEN the system SHALL CONTINUE TO wrap and align text within the block boundaries

3.5 WHEN labels with `^PW` (print width) are rendered THEN the system SHALL CONTINUE TO center the content within the full label width

3.6 WHEN reverse print (`^FR`) is applied to text THEN the system SHALL CONTINUE TO invert the text rendering correctly

3.7 WHEN golden tests for non-text elements (barcodes, graphic boxes, QR codes, etc.) are run THEN the system SHALL CONTINUE TO pass within the existing tolerance thresholds
